Key findings

Single-thread overhead: 14.35% (16.86M → 14.44M EPS, p < 0.001)
This is the baseline virtualization tax per core. On ARM64→ARM64 UTM there's no instruction translation, so this overhead comes purely from hypervisor scheduling — the vCPU periodically yields to the macOS host scheduler, and each re-entry into the VM carries a small but consistent cost. The extremely low CV of 0.38% on the VM side tells you the overhead is stable, not random noise.

Multi-thread overhead: 
38.73% → 14.66% — this is the critical correction. The previous number was misleading because it compared 10 bare-metal threads against 5 vCPUs. Now with 5 threads on both sides, the true virtualization overhead for parallel workloads is 14.66%, nearly identical to single-thread (12.63%). This is actually a stronger thesis finding — it shows that overhead per core is consistent regardless of thread count, meaning the hypervisor scheduling cost scales linearly rather than compounding.

Bare metal CPU time split changed significantly:
83.5% → 52.8% user, 15.1% → 2.8% sys, 1.4% → 44.6% idle — this is because with 5 threads on a 10-core machine, the other 5 cores are idle, so the system-wide top average now includes 5 idle cores dragging down user% and sys%. The VM shows 0% idle as before since all 5 vCPUs are fully loaded. For your report, note that the CPU split comparison is most meaningful for the VM (all cores loaded) and should be interpreted carefully for bare metal in the 5T run.

Scaling efficiency now comparable: 
BM 86.8% vs VM 84.8% (+2.0 pts) — previously bare metal appeared worse than VM at scaling (59.3% vs 84.8%) because bare metal was spreading work across mixed P/E cores. Now with both constrained to 5 threads, bare metal uses only its 5 P-cores and scales almost identically to the VM. The 2-point difference is within noise.
